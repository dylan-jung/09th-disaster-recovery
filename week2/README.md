## 📝 Runbook 작성 가이드라인 (or 자유 형식)

### 장애 대응 플로우 런북

> **런북(Runbook)**  
> 실제 운영 환경에서 긴급 상황 발생 시 빠르게 대응하기 위한 절차 문서

**장애 / 에러가 발생했을 때 어떻게 대응할 것인가?**

- 실제 경험한 상황
- 가상의 시나리오  
  (ex. 애플리케이션, 인프라, 데이터베이스 레벨 등)


## 1️⃣ 장애 시나리오

어떤 장애 상황인지 설명해주세요.

- 어떤 서비스 / 환경에서 발생?


## 2️⃣ 장애 인지

어떻게 이 장애를 인지하게 되었는지 설명해주세요.

예시

- Grafana alert
- Slack 알람
- 유저 문의

가능하면 **어떤 지표를 보고 판단했는지도 작성해주세요.**


## 3️⃣ 대응 플로우

장애를 인지한 후 **처음에 무엇을 확인했는지 / 확인할지** 작성해주세요.

> **처음에 어느 지점을 확인할 것인가?**

### 예시 (간단 ver)

```kotlin
1. Grafana Dashboard 확인
   - latency
   - error rate
   - traffic

2. 최근 배포 여부 확인
   - ArgoCD / CI 로그 확인

3. 로그 검색
   - Loki / ELK

4. Pod 상태 확인
   kubectl get pods

5. DB 상태 확인
   connection pool / slow query
```
### 4️⃣ 초기 대응

장애를 빠르게 완화하기 위한 방법은 무엇이었나요?
예시
- Rollback
- Pod restart
- Auto scaling
- Feature flag off

### 5️⃣ 원인 분석

장애 원인을 작성해주세요.

예시
- 메모리 누수
- 코드 버그
- API timeout

### 6️⃣ 사후 방지 대책 🚀

같은 장애가 다시 발생하지 않도록 어떻게 대응할 건가요?

혹은 장애를 더 빨리 인지하기 위해 어떤 체계를 만들 건가요?

예시
- SLO 기반 alert 설정
- Circuit breaker
- 로그 추가
- Dashboard 개선
